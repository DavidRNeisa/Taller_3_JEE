Guía de selección de estructuras

¿Necesitas...
│
├── Acceso por posición rápido (O(1)) → Arreglo / vector
├── Inserción/eliminación rápida en el medio → Lista enlazada
├── LIFO (deshacer, parsing) → Pila
├── FIFO (colas de espera, BFS) → Cola
├── Prioridad (evento más urgente) → Heap / priority_queue
├── Búsqueda por clave O(1) → Tabla hash / unordered_map
├── Búsqueda ordenada O(log n) → BST / map
├── Datos siempre ordenados con O(log n) garantizado → AVL / Red-Black
├── Comprimir datos → Árbol de Huffman
├── Relaciones entre entidades → Grafo
│   ├── Camino más corto (sin pesos) → BFS
│   ├── Camino más corto (pesos ≥ 0, un origen) → Dijkstra
│   ├── Camino más corto (todos los pares) → Floyd-Warshall
│   ├── Árbol de expansión mínima → Prim / Kruskal
│   └── Detección de ciclos / componentes → DFS
└── Búsqueda espacial 2D → Quadtree / K-d Tree

Reflexión final: las tres preguntas del ingeniero

  1. ¿Qué operaciones son más frecuentes? Diseña alrededor de la operación más común.
  2. ¿Cuántos datos habrá? La diferencia entre O(n) y O(n²) es irrelevante para n=100; es vital para n=10⁶.
  3. ¿Qué garantías necesito? ¿O(1) amortizado es suficiente, o necesito O(1) garantizado en cada operación?
La lección central del curso: No existe una "mejor" estructura de datos. Existe la estructura más adecuada para el problema específico. Un buen ingeniero conoce el repertorio completo y sabe cuándo usar cada herramienta.

Conexiones entre estructuras

  • Una pila implementa el recorrido DFS en grafos.
  • Una cola implementa el recorrido BFS en grafos.
  • Un heap acelera Dijkstra y Prim.
  • Un árbol es un grafo conexo acíclico.
  • La tabla hash permite la memoización que transforma recursión exponencial en lineal.
  • El Union-Find (árbol de conjuntos disjuntos) es esencial para Kruskal.

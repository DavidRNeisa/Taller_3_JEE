Clase 12: Proyecto — Entrega 1

Plantear, documentar y diseñar la solución antes de codificar

¿Qué se entrega?

  1. Descripción clara del problema
  2. Justificación de las estructuras de datos elegidas
  3. Casos de uso con operaciones y complejidades esperadas
  4. Pseudocódigo de las estructuras y operaciones principales
  5. Plan de implementación (cronograma de avance)

Ejemplo guiado: Sistema de urgencias hospitalarias

Descripción

Los pacientes llegan a urgencias con niveles de prioridad (crítico=4, urgente=3, moderado=2, leve=1). Dentro del mismo nivel se atiende por orden de llegada.

Justificación de estructuras

NecesidadEstructuraJustificación
Atención por prioridadCola de prioridad (Heap)O(log n) insertar/extraer más urgente
Historial de pacientesLista enlazadaInserción dinámica sin límite
Búsqueda por IDTabla hashO(1) promedio de acceso

Pseudocódigo del diseño

ESTRUCTURA Paciente:
  id: entero
  nombre: cadena
  urgencia: entero  // 1–4
  horaLlegada: entero

SISTEMA Hospital:
  colaUrgencias: ColaPrioridad<Paciente>
  historial:     TablaHash<id, Paciente>
  listaTotal:    Lista<Paciente>

  FUNCIÓN registrar(p):
    colaUrgencias.insertar(p)   // O(log n)
    historial.insertar(p.id, p) // O(1)
    listaTotal.agregar(p)       // O(1) con puntero a cola

  FUNCIÓN atenderSiguiente() → Paciente:
    RETORNAR colaUrgencias.extraerMax() // O(log n)

  FUNCIÓN buscarPaciente(id) → Paciente:
    RETORNAR historial.buscar(id)       // O(1)

Errores comunes en el diseño

  • Elegir una estructura sin justificar la complejidad de la operación principal.
  • Ignorar los casos borde (lista vacía, duplicados, datos inválidos).
  • No estimar el volumen de datos esperado.
  • Diseñar sin pensar en qué operaciones serán más frecuentes.
Principio de diseño: elige la estructura que minimiza la complejidad de la operación más frecuente.
